相机 UI和实现 方案：拍照/录像切换 + 单/双画面切换 + 横竖屏拍摄 + 全量设置入口

本版把需求拆成 4 个明确状态：单画面竖屏、单画面横屏、双画面竖屏、双画面横屏，并在主界面补充“拍照 / 录像”一级切换。这里的双画面特指同一摄像头输出两种方向构图的实时预览，不是双摄。单画面模式允许用户只拍一个画面或单独录像；双画面模式下，主画面与副画面都保持实时取景，并且横竖屏都能各自拍照 / 录像。设置按钮补到主界面顶部，进入后可承接系统相机常用设置项，并支持按设备能力动态显隐。

模式：单画面 / 双画面
方向：横屏 / 竖屏
能力：主副画面拍照 / 录像切换
设置：承接系统 Camera 能力

模式矩阵

建议把一级入口拆成两层：第一层控制“拍照 / 录像”，第二层控制“单画面 / 双画面”，再由系统姿态或用户手动按钮决定横竖屏呈现。这样规则清晰，避免把“拍摄类型”“单双画面”和“横竖方向”混成一个入口。
拍照 / 录像切换一级常驻入口，拍照与录像共用同一套单/双画面与横竖屏规则。
单画面模式只保留一个取景流，支持竖拍或横拍，也支持单独录像。
双画面模式同一摄像头下主副画面同时实时预览，两个画面都可独立拍照 / 录像。
设置入口顶部齿轮进入参数面板，功能按设备能力映射。

拍照 / 录像切换

一级常驻入口，切换后只改变输出类型，不改变当前单/双画面与横竖屏状态。

主快门 / 主录制

建议交互流（补充拍照 / 录像切换）

1

默认进入拍照 · 单画面

降低首次学习成本，首开即能拍照；上方常驻切换可一键进入录像。

2

点击切到录像或双画面

先切换拍照 / 录像，再决定是否进入双画面；双画面下出现副画面与对应录制能力。

3

横竖屏自动适配

竖屏时主竖副横；横屏时主横副竖，也允许手动交换；拍照与录像两态都共用这套规则。

4

齿轮进入完整设置

按设备能力展示系统相机可用功能、录像参数，以及同摄双画面扩展项。

实现层建议：
1. 单画面模式只维持一个预览 Surface / Texture，减少功耗与发热。
2. 双画面模式主副都来自同一摄像头，建议共享同一 Camera session 与同源预览纹理，再分别做横竖构图与裁剪。
3. 设置页不要写死“全部可用”，应按 Camera 能力探测返回结果动态展示，否则会和真实设备能力冲突。
4. 双画面拍摄建议定义清楚：拍照态下主快门拍主图、副快门拍副图；录像态下主录制 / 副录制 / 同时录制的规则也应单独开关。
技术可行性分析（基于当前版本：单摄像头 + 拍照/录像切换 + 横竖双画面）
当前方案已经明确限定为：双画面仅指同一摄像头的同源预览复用，不是双摄、不是前后同开；同时新增拍照 / 录像一级切换。其技术模型可抽象为：
Camera（单一流） ↓ 共享帧 / GPU Buffer ↓ 渲染层 ├─ 主画面（当前方向） ├─ 副画面（另一方向） ↓ 输出层 ├─ 拍照：横图 16:9 / 竖图 9:16 / 方图 1:1 └─ 录像：横视频 16:9 / 竖视频 9:16
可行性结论
该方案在 Android、iOS、鸿蒙上均具备工程可行性。核心不是“双流拍摄”，而是单流多视图 + 多比例裁剪 / 录制编排。主副画面只是在 UI 层切换展示权重，不需要切换摄像头；拍照与录像共用同一预览链路。
推荐实现原则
底层尽量保留更大的原始画幅（优先 4:3 或 full sensor frame），在预览层实时生成横图 16:9、竖图 9:16 与方图 1:1 的构图框；拍照时由同一帧统一裁剪输出，录像时复用同一预览链路进入对应编码与保存流程。
平台评估
Android
Camera2 / CameraX 可稳定支持单摄像头预览流复用。可通过 SurfaceTexture、TextureView 或 OpenGL 进行主副画面渲染。实现成本最低，最适合作为首发平台。
iOS
基于 AVFoundation 可行，但更推荐使用 AVCaptureVideoDataOutput + Metal / CoreImage 做双画面自绘。不能依赖系统相机 UI，需要自定义渲染与构图逻辑。
鸿蒙
基于 Camera Kit 可实现同一摄像头的多视图预览与裁剪输出，整体实现思路接近 Android，但具体可调参数与性能表现更依赖设备能力。
关键问题：竖屏 9:16 为什么左右会“漏很多”
这是由纵横比差异决定的正常现象，并非相机异常。横屏 16:9 是宽画幅，竖屏 9:16 是窄画幅；在同一张原始画面中，竖屏要成立，就必须裁掉更多左右区域。因此问题不在于“如何不裁”，而在于如何让用户提前看见裁切边界，并尽量减少信息损失。
推荐解决策略
1）底层保留更大源图，不要只保留已经裁成 16:9 的预览流。
2）竖拍时显示真实 9:16 成片框，并对左右非成片区做轻遮罩。
3）保留横图小窗，让用户同时感知横向完整构图。
4）拍照后基于同一张原始图统一裁出横图 / 竖图 / 方图，保证曝光与时间戳一致。
不建议的方案
1）不要尝试用两个独立拍照流分别产出横图和竖图。
2）不要把竖图硬拉到和横图一样宽，这会造成畸变。
3）不要在拍完后才告知用户被裁切，构图边界必须在预览时就可见。
关于 1:1（16:16）方图
如果底层保留的是更大的原始画面（例如 4:3），则同一张图可以同时裁出 16:9 / 9:16 / 1:1 三种比例。工程上应直接将“16:16”定义为 1:1 方图，并与横图、竖图共享同一套裁剪链路。
